# AdMob eCPM 急落の調査と修正指示書

- 日付: 2026-06-19
- 起票: eCPM が 2 週間で約 ¥100 → ¥5 に急落（バグ疑い）との報告
- ステータス: **調査完了 / コード未修正（別タスク進行中のため着手保留）**
- 対象リポ: `tsurilog-native`（広告は native 側のみ。backend は無関係）
- 調査方針: READ-ONLY。コードは変更していない。

---

## 0. TL;DR（結論を先に）

1. **「¥100 → ¥5」の大半は、おそらく新規バグではない。** 7 日間で収益 ¥5・表示 966 回という極小母数で、eCPM は統計ノイズに支配される。表示回数が 2.5 倍に増えた結果、バナー本来の低単価へ平均回帰した、という説明が最も自然。
2. **広告コードは 2026-04-18 以降変更なし**。直近 2 週間に広告関連のコード変更・新規設置はゼロ。リクエスト/表示の急増は 6 月初旬の大量リリース（強制アップデートゲート・AI 機能・初心者向け機能）による利用増で説明できる。
3. **ただし eCPM を構造的に下げている実害のある設定は実在する**（下記 §3）。これは「2 週間の急落原因」ではなく「恒常的な天井の低さ」。修正価値はある。
4. **真因確定は AdMob 管理画面が先**（§2）。無効トラフィック/視認性/フォーマット別 eCPM はコードからは分からない。コード修正はその後。

---

## 1. 観測データ（管理画面スクショ 2026-06-19 / 過去 7 日 vs その前 7 日）

| 指標 | 今期間 | 前期間（逆算） | 変化 |
|---|---|---|---|
| 推定収益 | ¥5 | ¥17 | -69.90% |
| リクエスト | 1,460 | ~435 | +235.63% |
| 表示回数 | 966 | ~390 | +147.69% |
| マッチ率 | 75.34% | ~96% | -21.90% |
| eCPM | ¥5 | ~¥42 | -87.85% |

- 月次: 今月 ¥26 / 先月 ¥85。
- 検算: ¥5 ÷ 966 × 1000 = eCPM ¥5.2 ✓ / ¥17 ÷ 390 × 1000 = eCPM ¥43.6 ✓ → 数値は内部整合。

### データの解釈（重要）

- **母数が極小**。966 表示・収益 ¥5。この規模では eCPM の分散が非常に大きく、単発の高単価インプレで「¥100」は容易に出る。「異常な急落」に見えるのは **分母（表示回数）の急増による見かけ上の希釈**。
- リクエスト急増（+235%）＋マッチ率低下（-21.9%）は、利用が急に増えたときの典型挙動（Google が増えた在庫を全部は埋めない）。
- ¥5 という絶対値の低さは「非パーソナライズ・バナーのみ」という構成の天井（§3-1, §3-2）とも整合する。**無効トラフィック判定の可能性も残る**ため §2 で要確認。

## 2. 【最優先】まず AdMob 管理画面で確認すること（コード修正の前）

コードだけでは真因を確定できない。以下を管理画面で確認し、原因が「コード」か「アカウント/設定」かを切り分ける。

- [ ] **ポリシーセンター / アカウント**: 無効なトラフィック警告・ポリシー違反・支払い保留がないか。¥5 の異常低値はこれの可能性あり（その場合コード修正では直らない）。
- [ ] **eCPM をディメンション分解**: 「広告フォーマット別」「国/地域別」「日別」。どこで落ちたかを特定（バナー単価そのものか、特定国の流入増か）。
- [ ] **視認性（viewability）率**: 低ければ §3-3（absolute 配置で覆われている）を裏付け。
- [ ] **app-ads.txt** の検証ステータス（未検証だと需要が減る）。
- [ ] 期間を「過去 30〜60 日」に広げ、本当に 2 週間前まで ¥100 水準が継続していたか確認（単発スパイクでないか）。

> ここで「無効トラフィック/ポリシー/支払い」が原因と判明した場合は、以下のコード修正では解決しない。Google への対応（トラフィック品質改善・異議申し立て）が必要。

## 3. コード調査で判明した eCPM 抑制要因（実害あり・修正対象）

すべて `tsurilog-native` 内。広告は banner のみが本番稼働、rewarded / native は未使用デッドコード。

### 3-1.【最重要】非パーソナライズ広告を全ユーザーに強制 + 同意フロー皆無

- 箇所: `components/ads/banner-ad.tsx:43`、`components/ads/rewarded-ad-modal.tsx:58`、`components/ads/native-ad-item.tsx:25`
  ```ts
  requestOptions={{ requestNonPersonalizedAdsOnly: true }}
  ```
- `mobileAds()` 初期化・UMP（User Messaging Platform / `AdsConsent`）同意取得が**コード上に一切ない**（grep 済み）。
- **影響**: 非パーソナライズ広告は単価が構造的に低い（一般に -30〜50%、市場により更に）。eCPM の天井を恒常的に下げている。これは 2 週間の急落原因ではなく「ずっと低い理由」。
- 2025-12 のファイル作成時から存在 → 今回の急落とは無関係だが、**収益改善の最大レバー**。

### 3-2. 配信中は実質バナーのみ。高単価フォーマット（rewarded）が死んでいる

- `components/ads/rewarded-ad-modal.tsx` / `native-ad-item.tsx` はどこからも import されていない（参照 0 件）。
- **影響**: 収益上限がバナー単価に固定。rewarded（動画リワード）は本来 eCPM が桁違いに高い。

### 3-3. バナーが `position:absolute` 下部固定 → 視認性低下リスク

- `components/ads/banner-ad.tsx:56-62`（`position:"absolute"`, `zIndex:1000`, `bottom`）。
- **影響(推測)**: 他 UI/モーダル/タブバーに覆われると viewability が落ち、入札単価が下がる。§2 の viewability 率で要確認。

### 3-4. map のバナー再マウント + key なし → 無効/低品質インプレ水増し

- `app/(tabs)/map.tsx:662`: `{!isPlacingPoint && <BannerAd ... />}`
- 新規ポイント登録の開始/キャンセルのたびにアンマウント→再マウント→新規広告リクエスト。`key` 未指定で再生成が不安定。
- **影響**: 低品質インプレを増やし eCPM をさらに悪化させる方向（収益増にはほぼ寄与しない）。マップはタブで利用頻度が高く、リクエスト急増（+235%）の一因の可能性。

## 4. 修正指示書（別タスク完了後に着手 / 上から優先）

> 前提: §2 の管理画面確認を先に実施し、「無効トラフィック/ポリシー」起因でないことを確認してから着手する。
> ブランチ運用: `tsurilog-native` で `develop` 起点に `feature/admob-ecpm-fix` を切る（CLAUDE.md §6.1）。UI を触るので最後に `/native-qa`。広告コードに触れるので影響範囲を PR 本文に明記。

### Step 1 — 同意管理(UMP)を導入し、同意ユーザーにはパーソナライズ広告を配信（§3-1）★最大効果
- アプリ起動時に `mobileAds()` 初期化 + `AdsConsent`（UMP）で同意取得フローを実装。
- `requestNonPersonalizedAdsOnly` を **固定 true をやめ、同意ステータスに連動**させる（同意あり→パーソナライズ配信）。
- iOS は ATT（App Tracking Transparency）との整合も確認。
- 注意: AdMob 管理画面側の GDPR/同意メッセージ設定も必要。コードだけでは完結しない。

### Step 2 — viewability 改善（§3-3）
- §2 で viewability が低いと判明した場合のみ。`position:absolute` で覆われていないか実機確認し、必要ならレイアウトに組み込む（absolute をやめる/被り回避）。

### Step 3 — map バナーの再マウント抑止（§3-4）
- `{!isPlacingPoint && <BannerAd>}` の出し分けをやめ、常時マウントして可視性のみ切替（`display:'none'` 等）or 安定 `key` 付与。低品質インプレを減らす。

### Step 4 — 高単価フォーマットの再活性化（§3-2）任意・要プロダクト判断
- rewarded を「AI アドバイス回数の追加」等のインセンティブ動線で復活させる、もしくは使わないなら削除。
- ※ UX に直結するためエンジニアリング単独で決めない。プロダクト側と合意の上で。

### やってはいけないこと
- **eCPM の数値だけを見て焦って広告を増やさない**。低品質インプレ増は eCPM をさらに下げ、無効トラフィック判定のリスクを上げる。
- 本番広告 ID をテストビルドに混在させない（現状の `__DEV__` / variant 出し分けは正しい。維持する）。

## 5. 率直な所見（同調しすぎない）

- 「明らかにバグ」という初期仮説には**慎重**であるべき。広告コードは 2 週間変更されておらず、母数が極小なため eCPM 単体の急落は**統計的に普通に起こりうる**。まず管理画面で実数の内訳を見るのが先。
- 一方で、§3-1（NPA 強制＋同意フロー皆無）は**確実に効いている恒常的な損失**で、急落とは別に直す価値が高い。「急落の犯人探し」より「収益構造の底上げ」として捉えるのが妥当。
- 母数が小さいうちは eCPM ではなく**絶対収益額・DAU あたり収益**で評価する方が判断を誤りにくい。

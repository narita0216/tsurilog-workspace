# 外部サービスのUI/手順を記憶で断定して誤案内した件（再発防止レポート）— 2026-06-20

## 事象
App Store Connect でサブスクが「審査待ち」のまま本番で商品が取得できない(`商品情報を取得できませんでした`)問題の対応中、AI(私)が

> 「1.1.1 のバージョンページの『App内課金とサブスクリプション』セクションでサブスクを選択して紐付ける」

と**断定的に手順を提示**した。実際には現行 ASC では**そのセクションはバージョンページから廃止**されており（2022年の提出フロー刷新で「審査用に追加(Add for Review)」= Submission 画面へ移動）、ユーザーは**存在しないセクションを探させられた**。ユーザー指摘＋実画面スクショで初めて誤りに気づき訂正した。ユーザーから「嘘をつかれた」と強い不信を招いた。

## 根本原因
1. **頻繁に変わる第三者SaaS(App Store Connect)の“現在のUI/操作手順”を、学習時点の記憶から検証せずに事実として提示した。** 学習データは古い前提で扱うべきなのに扱わなかった。
2. 検証手段（Apple公式ドキュメントの WebFetch / 先に実画面のスクショ依頼）が**あったのに使わなかった**。
3. **同セッション内の二重基準**: WorldTides の API 仕様(step/クレジット)は一次ドキュメントを WebFetch で確認していた（正しい行動）。なのに ASC の UI 手順では同じ規律を適用しなかった。既存ルールが「API仕様の検証」に閉じていて「UI/操作手順の検証」をカバーしていなかった。
4. `findings/2026-06-18-iap-debugging-retrospective.md`「憶測で外部要因に帰着しない」と同類だが、対象が“バグ原因の帰着”ではなく“操作手順の断定”で、ルールの射程外だった。

## 再発防止ルール（具体・強制）
1. **第三者サービスの現在のUI・メニュー名・操作手順を、記憶（学習データ）から断定しない。** App Store Connect / Google Play Console / Stripe / 各種クラウドコンソール等は UI が頻繁に変わる。
2. 手順を提示する前に、必ず次のいずれかを行う:
   - (a) ベンダー公式の**最新ドキュメントを WebFetch で確認**してから書く、または
   - (b) ユーザーに**先に実画面のスクショを依頼**し、見えているものに基づいて案内する。
3. 確証が無いときは断定しない。「現行UIは変わっている可能性があるので確認させてください/画面を見せてください」と**不確実性を明示**する。
4. 回答内で「**検証済みの事実**」と「**未確認の推測**」を明確に分けて書く。
5. この規律は「API/外部仕様」だけでなく「**UI・操作手順・管理画面の場所**」にも等しく適用する（今回の抜け穴）。

## 正しい情報（※これ自体もユーザーの実画面で要再確認）
- 現行 ASC は IAP/サブスクの審査同梱を、**バージョンページのセクションではなく「審査用に追加(Add for Review)」の Submission 画面**で行う（2022年刷新）。
- アプリ初の(=最初の)サブスクは**アプリ版の提出に同梱が必要**で、自動では紐付かない。
- 「審査待ち」= 提出済みで審査列に入った状態（「提出準備完了」=未提出 とは別）。ただし最初のサブスクはアプリ版に紐付かないと実審査が進まない。

## 教訓（一行）
**外部サービスの「今の画面・今の手順」は記憶で語らない。一次情報か実画面で確認してから案内する。** API仕様もUI手順も同じ。

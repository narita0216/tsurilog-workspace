# 「count」出力でツール呼び出しが失敗し処理が止まる問題の原因調査

日付: 2026-06-12
症状: ターン途中で `count` という謎のテキストが出力され、「ツール呼び出しに失敗して停止しました」エラーで処理が止まる。6/8 頃から頻発。

## 結論

**設定・hook・キャッシュは無関係。** 原因は「6/2 から resume し続けた単一の巨大セッション(transcript 49MB / assistant メッセージ 5,300 件)で、モデルがツール呼び出しを専用形式ではなくプレーンテキストとして出力する故障が起き、その失敗テキストが履歴に残って自己増殖した」こと。

## 証拠(transcript: `~/.claude/projects/.../4c70e5fb-*.jsonl`)

- 失敗ターンの実体: assistant の text ブロックに以下が**そのまま文字列として**出力されている。

  ```
  count
  <invoke name="Read">
  <parameter name="file_path">...</parameter>
  </invoke>
  ```

  本来ツール呼び出しは専用ブロックで出力されるが、テキストに漏れたためハーネスがパースできず「ツール呼び出し失敗」になる。`count` は壊れた呼び出し構文の断片。

- 発生は **33 件すべて `claude-opus-4-8`**(当時のセッションモデル)。6/8 に初発 → 6/9 に 7 件 → 6/11 は 13:00 前後にほぼ毎ターン失敗(2〜3 分間隔で 20 件)と**加速**。
- 加速の理由: 一度漏れた `count\n<invoke ...` テキストが会話履歴(と要約)に残り、モデルがそれを「前例」として模倣する**コンテキスト汚染**。セッションを続けるほど悪化する。
- 6/11 に新規開始した別セッション(`ed6bd089`、claude-fable-5)では **発生 0 件**。
- hook は全数確認済み: workspace の `workspace-sync.sh` / `contract-check.sh` / `php-lint.sh` / `harness-autosave.sh`、グローバルの gstack / SUPERSET notify / osascript 通知のいずれも `count` を出力するものは無い。
- Claude Code は障害発生時 2.1.169 → 6/11 に 2.1.173 へ自動更新済み。

## 対処

1. **汚染セッション(4c70e5fb)を resume しない。** 新しいセッションで作業する(これだけで解消する。実際、新セッションでは再発していない)。
2. セッションを何日も resume し続けない。区切りごとに新セッション(/clear)。長期文脈は memory / findings / CLAUDE.md に書き出してから切る(本ワークスペースの運用方針とも一致)。
3. 再発したら: その場で続けて打鍵せず、いったん新セッションへ。失敗テキストが履歴に入るほど再発率が上がる。

## 学び

- 「ツール呼び出し失敗で停止」が**同一セッション内で頻発し始めたら、設定ではなくセッション汚染を疑う**。transcript(`~/.claude/projects/<project>/<session>.jsonl`)を grep すれば失敗の実体が見える。
- 調査コマンド例: `grep -n '<invoke name=' <session>.jsonl`(漏れたツール呼び出しテキストの検出)、`grep -o '"stop_reason":"[^"]*"' | sort | uniq -c`。

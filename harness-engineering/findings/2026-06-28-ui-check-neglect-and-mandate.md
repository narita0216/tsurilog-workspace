# 2026-06-28 UIチェックを怠った事故と再発防止(義務化)

1.1.4 のUI改修で、ユーザーから「全体的にUIがひどい」と指摘される品質で出してしまった。
原因と再発防止を記録し、以後の開発でUIチェックを**義務**とする。

## 何が起きたか(実害)
- **有料の広告非表示**: `BannerAd` を「課金者は `null` を返す」だけにし、各画面が広告用に
  確保していた**余白(マジックナンバー)を残した**。結果、ホーム/釣行記録一覧/詳細に謎の下余白、
  マップは釣り開始/終了ボタンが下に詰まらず変な余白。
- **釣果カードの文字見切れ**: カード幅が文脈ごとに違う(図鑑320 / 詳細モーダル ~300〜340)のに、
  1サイズ(図鑑320)でしか確認せず「OK」と一般化。No. やメタ行が右端で切れていた。
- **図鑑の日付フィルタが崩れ**: 横スクロールViewに `style`(flexGrow:0等)を付けず、flexで潰れた。
- 上記いずれも **lint/tsc は通っていた**。型と整形が通る ≠ 画面が正しい。

## 根本原因
1. **「広告を消す」を“非表示にする”だけのタスクと誤認**。広告は各画面のレイアウトと
   余白(reserve)で密結合していたのに、結合先を洗い出さなかった。
2. **1コンポーネント・1サイズ・1状態でのQAを全体に一般化**。共有コンポーネント(カード)を
   呼び出し箇所ごと/サイズごとに見ず、課金/無課金の両状態も確認しなかった。
3. **完了判定を tsc/eslint と単発スクショに依存**。「per-screen / per-state」での目視確認を省いた。
4. チェックの観点が暗黙知で、**強制するチェックリストが無かった**。

## 再発防止 — UI変更チェックリスト(義務・`/native-qa` で実施)
UIに触れる変更は、**マージ/完了report の前に必ず**以下を満たす。満たせない項目は理由を明記する。

- [ ] **影響画面を列挙**してから着手する(共有コンポーネントは grep で全呼び出し箇所を洗う)。
- [ ] **各影響画面を実機/シミュレータで目視**(スクショ添付)。1画面の確認を他画面へ一般化しない。
- [ ] **状態の網羅**: 無課金/有料、空/データあり、ローディング/エラー、最小〜大きめ幅。広告に関わる
      変更は **無課金=広告あり / 有料=広告なし の両方**を必ず撮る。
- [ ] **要素を消す/隠す時は、その要素が占めていた領域(余白/オフセット/reserve)も必ず詰める**。
      「表示が消えた」だけで終わらせない(phantom padding を残さない)。
- [ ] **共有コンポーネントは呼び出し箇所ごとの実寸**で確認(カードは図鑑/詳細/done/共有モーダル)。
- [ ] **文字の見切れ・折り返し・潰れ**を確認(長い魚種名、`numberOfLines`/`flexShrink`/`space-between`)。
- [ ] **横スクロール/リスト**は `style` の flex 指定(flexGrow:0 等)で潰れ・余剰拡大が無いか確認。
- [ ] tsc/eslint が通っただけで「UI OK」としない。

## 仕組み化
- `CLAUDE.md` §8.3 に本チェックリストへの参照を追加し、UI変更時の必須手順とする。
- 余白の付け外しは、画面ごとのマジックナンバーを直接 reserve に足すのではなく、
  `useIsPaid()` 等の単一判定でゲートする(今回 `hooks/use-is-paid.ts` を新設)。

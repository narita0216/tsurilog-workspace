# 釣れる度 再設計 確定要件（品質最優先・一般主体＋データ自動成長）

> **位置づけ**: ピボット(ADR-0011)A2/A3 の実装要件（確定版）。下書き要件のレビューで判明した矛盾を解消し、品質最優先で確定したもの。
> **前提**: 現行仕様 `assessment/condition-score-current-spec.md`。親 initiative `2026-06-pivot-condition-score-monetization.md`。
> **方針**: 魚種非依存・一般知識ベース・**絶対値スコア(0〜100)**・時刻ごと(時間軸グラフ)。**今は釣行データを最小影響にし、データが増えるほど自動で影響を上げる**。

---

## 0. 確定した設計判断（レビュー反映）

| 論点 | 確定 | 理由 |
|---|---|---|
| スコアの性質 | **絶対値 0〜100** | 一般重みが床になり潰れない |
| general と data の軸 | **両方とも「絶対の良さ 0〜1」** | 同軸で混ぜる。偏差(0.5中心)は使わない |
| data の作り方 | **prior→posterior の絶対 goodness**（rel・baseline・FINAL_GAIN は**廃止**） | データが増えても50へドリフトしない（下書きの矛盾を解消） |
| 較正の範囲 | **グローバル（n_i = 全記録 × 条件値）** | 記録≒2人でスポット較正は不可能。環境9値は地点固有なのでスコアの地点差は保たれる |
| データの重み | **w_i = n_i/(n_i + K_shrink)、K_shrink 大** | 今は w≈0=一般主体。記録増で自動上昇＝要望そのもの |
| 集約 | **9条件の重み付き和（Σweight=100）** | 0〜100 に直接スケール |
| is_show | **撤廃（常に表示）** | 一般が床なので潰れない |

> **環境9値（地点・時刻固有）と較正（全国データ）は別物**。「スコアが地点で変わる」のは環境値が地点固有だから。較正を全国にしても地点差は出る。

## 1. スコア合成（確定式）

時刻 t・条件 i ごと:
```
general_i(t) ∈ [0,1]      # 一般スコア（§3 の連続関数・config駆動）
data_i(t)    ∈ [0,1]      # データの“絶対の良さ”（§2・prior→posterior）
w_i          = n_i / (n_i + K_shrink)      # n_i = 全記録で条件値が一致するログ件数
blended_i(t) = (1 - w_i) * general_i(t) + w_i * data_i(t)

all_rate(t)  = Σ_i ( blended_i(t) * weight_i )      # Σweight=100 → 0〜100
```
- **n_i=0 → w_i=0 → 純・一般**（潰れない）。**n_i 増 → w_i 自動上昇 → データが効き始める**。
- **K_shrink は大きめ(config・初期 ~2000)**。現状 n_i は1桁→ w≈0=ほぼ一般。条件ごとに n_i は別（天気は早く貯まり、気温(整数)は遅い＝**その条件のデータが十分になって初めて効く**＝正しい挙動）。

## 2. データ層（絶対 goodness・prior→posterior）

```
prior     = config の基準釣れ率（“典型的にこれくらい釣れる”の事前値。例 0.2）
p_i       = (caught + K_smooth * prior) / (total + K_smooth)    # 0〜1 の釣れ率（少数は prior へ縮約）
data_i    = goodness(p_i)                                       # 固定アンカー(config)で 0〜1 の“良さ”へ写像
                                                               # 例: p=0→0.0, p=prior→0.5, p=prior*2.5→1.0（単調・クランプ）
```
- **偏差(rel = s^G/(s^G+b^G)) は使わない。** baseline(走る平均)も FINAL_GAIN も廃止。
- アンカーが「走る平均」ではなく**固定値(config)**なので、データが増えても50中心に潰れない＝general と同じ絶対軸。
- `caught`/`total` は現行 `calculateRate` のログ集計を流用（`where('logs.<cond>', envValue)`、`is_caught`、fish_id 任意）。**スポット絞り(only_mesh_5km)はしない。**

- **時間軸グラフの構造**: 日次条件(月・潮回り・水温)＝全時刻一定＝グラフの高さ。時刻条件(他6)＝形。
- **連続化の品質効果**: バケット境界での段差が消え、隣接時刻のスコアが自然に繋がる。

### §8 で確認すべき差分（実装前にコードで照合）

- hour は現行が離散rating＋非対称マズメ窓 → **連続山型は新規実装**（日出日入計算は流用）。

## 5. 潮の動き（位相 × 振幅）

```
潮の動きスコア = 位相係数 × 振幅係数
位相係数: 満潮/干潮(潮止まり)=0.2 / 上げ3分・下げ3分=0.6 / 上げ7分・下げ7分=1.0   # tid_action_id(6位相)から
振幅係数: 釣行時刻に近い満潮水位−干潮水位を正規化（当面 絶対値、後でスポット相対）
```
- 掛け算なので「潮止まりだが干満差大」は低＝物理的に正しい。
- **潮回り(tid_type)は別条件として継続**（地力＝状態＋ラベル）。

### ⚠️ データ前提（確認済み・要・保存追加）
- WorldTides extremes は満潮/干潮の **height を返し取得済み**（`raw_tide_height_cm` の補間に使用）。だが **env_cache に保存しているのは時刻ごとの補間潮位だけで、「直近の満潮−干潮＝振幅」は未保存**。rate は env_cache を読むので**そのままでは振幅が引けない**。
- → **GetEnvData で当日の満干の高さ(or 振幅)を env_cache に保存する追加が必要**（データ源はある）。
- フォールバック: 振幅が取れない時は **振幅係数=1.0 固定（位相のみ）**。
- 絶対値正規化は**地域バイアス**（潮位差の小さい湾内が常に低）→ 早めにスポット相対化。

## 6. 表示
- `HourlyCatchRateChart`(時刻×0〜100折れ線)・`rateVerdict`(質的ラベル) を流用。
- **`rateVerdict` 閾値は絶対スケール用に再導出**。現行(偏差50中心 ≥68/56/44/32)は使わない。
- **インフレ防止（品質の要）**: general 初期値が高めに偏ると凡庸な日でも65〜75が出て「常に好調」＝情報量ゼロになる。**典型日≈50〜60 / 好調日80+ / 渋い日<40 に分布するよう general 初期値と閾値を較正**（実スコア分布を見て調整）。磁石の健全性指標(リテンション)で監視。
- is_show 撤廃に伴い「データ不足で非表示」分岐を削除（常に一般スコアを表示）。
- robust 維持（◎○△・方向性・潮回りラベル）。精密な符号分解は出さない。

## 7. チューニング定数・config（廃止/新設を明記）

| 定数 | 役割 | 扱い |
|---|---|---|
| `K_shrink` | 一般↔データ混合 n/(n+K) | **新設・大きめ(初期 ~2000)**。今は w≈0、データ増で自動上昇 |
| `K_smooth` | データ層内のベイズ縮約(prior へ) | 流用(8) |
| `prior` | データ層の事前釣れ率 | **新設(config・例0.2)** |
| `goodness()` アンカー | 釣れ率→0〜1の良さ | **新設(config)** |
| `G`(CONTRAST_G) | 偏差写像 | **廃止**（絶対 goodness に置換） |
| `FINAL_GAIN` | 偏差の50回帰引き戻し | **廃止**（重み付き和に置換） |
| `baseline`(走る平均) | 偏差の基準 | **廃止** |
| 各 general 関数(§3) | 条件→0〜1 | config・体感調整 |
| 重み(§4) | 条件間配分 | config・体感調整 |
| verdict 閾値 | スコア→ことば | 絶対スケール用に再導出 |
| 気圧 | 10個目 | 取得でき次第 config 追加 |

## 8. 実装順序（品質最優先）
1. **general 一本で骨格を作る**（9条件の連続関数＋重み付き和＝all_rate、0〜100、config駆動）。is_show 撤廃。verdict 閾値暫定。→ これだけで「一般知識ベースの絶対スコア」が完成。
2. **潮 位相×振幅**（env_cache に振幅保存を追加。取れなければ位相のみ）。
3. **データ層(shrinkage)を枠として接続**（prior→posterior の絶対 goodness、w=n/(n+K_shrink)、K大）。今は w≈0 なので体験は1とほぼ同じ＝**安全に“育つ枠”を仕込む**。
4. **較正**: 実スコア分布を見て general 初期値・重み・verdict 閾値を調整（インフレ防止）。リテンション(磁石健全性)を監視。

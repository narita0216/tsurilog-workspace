# 釣れる度 再設計 確定要件（品質最優先・一般主体＋データ自動成長）

> **位置づけ**: ピボット(ADR-0011)A2/A3 の実装要件（確定版）。下書き要件のレビューで判明した矛盾を解消し、品質最優先で確定したもの。
> **前提**: 現行仕様 `assessment/condition-score-current-spec.md`。親 initiative `2026-06-pivot-condition-score-monetization.md`。
> **方針**: 魚種非依存・一般知識ベース・**絶対値スコア(0〜100)**・時刻ごと(時間軸グラフ)。**今は釣行データを最小影響にし、データが増えるほど自動で影響を上げる**。

## 0. 確定した設計判断（レビュー反映）

| 論点 | 確定 | 理由 |
|---|---|---|
| スコアの性質 | **絶対値 0〜100** | 一般重みが床になり潰れない |
| general と data の軸 | **両方とも「絶対の良さ 0〜1」** | 同軸で混ぜる。偏差(0.5中心)は使わない |
| data の作り方 | **prior→posterior の絶対 goodness**（rel・baseline・FINAL_GAIN は**廃止**） | データが増えても50へドリフトしない（下書きの矛盾を解消） |
| 較正の範囲 | **グローバル（n_i = 全記録 × 条件値）** | 記録≒2人でスポット較正は不可能。環境9値は地点固有なのでスコアの地点差は保たれる |
| データの重み | **w_i = n_i/(n_i + K_shrink)、K_shrink 大** | 今は w≈0=一般主体。記録増で自動上昇＝要望そのもの |
| 集約 | **10条件の重み付き和（Σweight=100）** | 0〜100 に直接スケール |
| is_show | **撤廃（常に表示）** | 一般が床なので潰れない |
| 気圧 | **10番目の条件として追加（傾向ベース・general専用）** | 取得安い(同エンドポイント・追加req0)・釣り効果大。logsに列が無いのでデータ層はw≈0で安全 |

> **環境値（地点・時刻固有）と較正（全国データ）は別物**。「スコアが地点で変わる」のは環境値が地点固有だから。較正を全国にしても地点差は出る。

## 1. スコア合成（確定式）

時刻 t・条件 i ごと:
```
general_i(t) ∈ [0,1]      # 一般スコア（§3 の連続関数・config駆動）
data_i(t)    ∈ [0,1]      # データの“絶対の良さ”（§2・prior→posterior）
w_i          = n_i / (n_i + K_shrink)      # n_i = 全記録で条件値が一致するログ件数
blended_i(t) = (1 - w_i) * general_i(t) + w_i * data_i(t)

all_rate(t)  = Σ_i ( blended_i(t) * weight_i )      # i=1..10, Σweight=100 → 0〜100
```
- **n_i=0 → w_i=0 → 純・一般**（潰れない）。**n_i 増 → w_i 自動上昇 → データが効き始める**。
- **K_shrink は大きめ(config・初期 ~2000)**。現状 n_i は1桁→ w≈0=ほぼ一般。条件ごとに n_i は別（天気は早く貯まり、気温(整数)は遅い＝**その条件のデータが十分になって初めて効く**＝正しい挙動）。

## 2. データ層（絶対 goodness・prior→posterior）

```
prior     = config の基準釣れ率（“典型的にこれくらい釣れる”の事前値。例 0.2）
p_i       = (caught + K_smooth * prior) / (total + K_smooth)    # 0〜1 の釣れ率（少数は prior へ縮約）
data_i    = goodness(p_i)                                       # 固定アンカー(config)で 0〜1 の“良さ”へ写像
                                                               # 例: p=0→0.0, p=prior→0.5, p=prior*2.5→1.0（単調・クランプ）
```
- **偏差(rel = s^G/(s^G+b^G)) は使わない。** baseline(走る平均)も FINAL_GAIN も廃止。
- アンカーが「走る平均」ではなく**固定値(config)**なので、データが増えても50中心に潰れない＝general と同じ絶対軸。
- `caught`/`total` は現行 `calculateRate` のログ集計を流用（`where('logs.<cond>', envValue)`、`is_caught`、fish_id 任意）。**スポット絞り(only_mesh_5km)はしない。**

## 5. 潮の動き（位相 × 振幅）

```
潮の動きスコア(時刻H) = 位相係数(H) × 振幅係数(H)
位相係数: 満潮/干潮(潮止まり)=0.2 / 上げ3分・下げ3分=0.6 / 上げ7分・下げ7分=1.0   # tid_action_id(6位相)から
振幅係数: swing_amplitude(H) を正規化（当面 絶対値、後でスポット相対）

# ★振幅は「日次1値」ではなく時刻ごと（日潮不等を考慮）:
swing_amplitude(H) = | height(直後の極値) − height(直前の極値) |
                     # その時刻 H が挟まれている“今動いている干満ペア”の振幅。
                     # 同じ日でも午前の干満は激しく午後は緩い等(日潮不等)を自然に反映。
```
- 掛け算なので「潮止まりだが干満差大」は低＝物理的に正しい（大潮でも潮止まり/干満差小は低く出る）。
- **潮回り(tid_type)は採点しない**＝振幅がper時刻で大潮/小潮を内包（上位互換）。足し算で別計上すると潮止まりに一律ボーナスが乗り相互作用を壊す。**地力ラベルとして表示のみ**（§4・§6）。
- 振幅係数の初期アンカー: swing 0cm→0.2 / 50→0.5 / 100→0.75 / 150→0.9 / ≥200cm→1.0。位相係数: 潮止まり=0.2 / 上げ下げ3分=0.6 / 上げ下げ7分=1.0。

### ⚠️ データ前提（確認済み・要・保存追加）
- WorldTides extremes は各極値の **時刻＋高さ(height) を返し取得済み**（`raw_tide_height_cm` の補間に使用）。
- だが env_cache に保存しているのは**時刻ごとの補間潮位だけ**で、**振幅は未保存**。
- → **GetEnvData で時刻ごとに `raw_tide_swing_cm`（その時刻の干満ペアの振幅）を env_cache に保存する追加が必要**。
  - 計算は `interpolateTideHeightCm` が**既に毎時求めている prev/next 極値**を流用： `|next.height − prev.height|`。位相(tid_action)も同じ挟む極値由来なので**位相×振幅が同一ペア基準で一貫**する。
  - **「当日の代表振幅1値」は不可**（日潮不等を落とす）。必ず時刻ごと。
- フォールバック: 前の極値が無い時刻(早朝端)など swing が取れない時は **振幅係数=1.0 固定（位相のみ）**。
- 絶対値正規化は**地域バイアス**（潮位差の小さい湾内が常に低）→ 早めにスポット相対化。

## 5b. 気圧傾向（pressure_trend・10番目）

```
気圧傾向スコア(時刻H) = f( Δp(H) )                 # Δp = pressure_msl(H) − pressure_msl(H−3h)  [hPa]
f: 単調減少（下降ほど高い）。初期アンカー(config):
   Δp ≤ −3      → 1.0   (急降下＝活性↑)
   Δp = −1.5    → 0.8
   Δp =  0      → 0.55  (安定)
   Δp = +1.5    → 0.45
   Δp ≥ +3      → 0.35  (急上昇＝渋り)
```
- **効くのは絶対値でなく変化（傾向）。** `pressure_msl`(hPa, 海面更正＝地点・標高で比較可)を使う。`surface_pressure`は使わない。
- 取得: **Open-Meteo の陸 forecast/archive(既存エンドポイント)の `hourly` に `pressure_msl` を1語追加するだけ**＝追加リクエスト0。一次情報で変数名・単位(hPa)確認済み(2026-06-25)。
- **24時間分を一括取得済み**なので Δp(3時間差)は**後処理で算出＝追加API不要**。端(H<3)は前日終端 or 当日先頭で代用、取れなければ係数=0.55(中立)。
- **データ層は当面 w≈0**: `logs` に気圧カラムが無く `data_i` を学習できない。general専用で開始。列追加＋バックフィルは「データが意味を持ち始めてから」（§1の自動成長と整合）。
- env_cache 保存: `raw_pressure_msl_hpa`（時刻ごと）を追加。Δ算出は rate 側で可。
- ⚠️ **archive(過去日)での `pressure_msl` 提供は docs 未明記** → 実装時に1回叩いて確認。取れなければ forecast のみ対応＋過去日は中立。

## 6. 表示・出力較正（確定）

### 6.1 合算後アフィン（レンジを10〜100に広げる）
```
raw     = Σ_i ( blended_i(t) * weight_i )       # floor のため raw の底 ≈ 19、天井 = 100
display = clamp( 10 + (raw - 19) / (100 - 19) * 90 , 10, 100 )   # ≒ 10 + (raw-19)*1.11
```
- 加重平均は中央に圧縮しがち → **各条件 floor は自然なまま（§3）、最後に1回アフィンで底10へ**。floor を無理に下げてガタつかせない。
- `19 / 100 / 90` は config。calibration 段階(§8-4)で実分布を見て微調整。
- 目安: 全軸最悪→10 / 渋い日≈22 / 普通≈56 / 良い≈75 / 絶好≈96。

### 6.2 verdict（10〜100スケール）

|---|---|

| ≥64 | 🎣 良い |
| ≥46 | 🙂 ふつう |

- **インフレ防止**: アフィンでフルレンジを使うので「常に好調」を回避。磁石の健全性指標(リテンション)で監視。

## 8. 実装順序（品質最優先）
1. **general 一本で骨格を作る**（10条件の連続関数＋重み付き和＝all_rate、0〜100、config駆動）。is_show 撤廃。verdict 閾値暫定。→ これだけで「一般知識ベースの絶対スコア」が完成。
   - 気圧: OpenMeteoClient の `hourly` に `pressure_msl` 追加 → env_cache `raw_pressure_msl_hpa` → Δp(3h)→f で general 化。archive 提供を1回確認。
2. **潮 位相×振幅**（env_cache に振幅 `raw_tide_swing_cm` 保存を追加。取れなければ位相のみ）。**この段階で潮回りを採点から外す**（手順1の間は振幅未配線なので潮回りが暫定の潮強度シグナル＝移行措置）。
3. **データ層(shrinkage)を枠として接続**（prior→posterior の絶対 goodness、w=n/(n+K_shrink)、K大）。今は w≈0 なので体験は1とほぼ同じ＝**安全に“育つ枠”を仕込む**。
4. **較正**: 実スコア分布を見て general 初期値・重み・verdict 閾値を調整（インフレ防止）。リテンション(磁石健全性)を監視。

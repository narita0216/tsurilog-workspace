## 1. 魚種別釣れる度グラフに一般スコアを薄く重ねる（比較表示）

**目的**: 魚種スコアと一般スコアは形が似ていて、パッと見で差が分からない。同一グラフに一般を薄く重ねて「この魚種はこの時間帯が一般より効く」を視覚化する（有料の価値可視化）。

**現状 [確認済]**
- `GetAnalysisRateController` は各時刻 `rates = ConditionScoreCalculator::score(env, lat, lng, Δ気圧, effectiveFishId)` を返す。`effectiveFishId` は「有料 かつ 魚種指定」時のみ魚種ID、それ以外 null。
- `score()` の戻り: `{ all_rate: float, condition_rates: {tide_movement,time_of_day,water_temperature,month,...} }`。魚種指定時は `all_rate` が**魚種調整後**の総合スコアになる。
- native `analysis.tsx`: メイン折れ線 `HourlyCatchRateChart` は **1系列**（`normalizedRows.map(r => r.rates.all_rate)`）。`condition_rates` は◎○△のフィット指標にのみ使用。
- → 現状、魚種選択中は「魚種スコア1本」のみで、比較対象の一般スコアが返っていない。

**要件 [提案]**
- backend: **有料ユーザーが魚種を指定したとき、一般（fishId=null）と魚種（fishId=X）の両方の `all_rate` を各時刻で返す**。後方互換のため既存 `all_rate` は主表示（魚種）のまま、`all_rate_general`（一般）を各 row に追加。未課金/魚種未選択時は従来通り（general のみ）。
- native: `HourlyCatchRateChart` に **副系列（一般）** を渡せるようにし、魚種選択中は**魚種＝実線・強調色／一般＝薄い破線**で重ねる。凡例（「魚種 / 一般」）を小さく表示。
- 魚種未選択・未課金時は従来の1本表示（変更なし）。

**影響範囲**: `GetAnalysisRateController.php`（both系列算出）、`app/analysis.tsx`（副系列受け渡し）、`components/analysis/hourly-catch-rate-chart.tsx`（2系列描画）。契約変更 → `/contract-check` 必須。

**計測**: なし（表示のみ）。

**[決定 2026-07-02]** 魚種選択時は**常に**一般スコアを**薄い破線**で重ねる（比較用）。**トグルは無し**。

## 2. フリープランのピンお気に入り数を制限（有料は無制限）

**目的**: 有料の実利価値を1つ増やす（お気に入り無制限）。

**現状 [確認済]**
- `RegisterPinController` は `Pin::updateOrCreate([user_id,lat,lng], ...)` で**上限なし**。同一 lat/lng は更新。
- `planKey()`/`isPaid()` で有料判定可能。

**[決定 2026-07-02]** 本番pins集計（有効390件/135人・中央値2・p75=3・N=3で23%超過）を踏まえ決定:
- **フリー: 有効ピン最大 3 件**（`deleted_at` 空のみカウント。削除で空けば再追加可）。
- **有料: 無制限**。
- **既存の上限超過ユーザーはグランドファーザー**（今のピンは削除/非表示しない＝**新規追加のみ 422 でブロック**）。
- 「便利さ」課金として conversion を取りにいく狙い（doctrine「売るのは深さ・便利さ」と整合）。magnet 本体（無料の一般釣れる度）は別途無料維持。

**要件 [提案]**
- 上限到達後の**新規**ピン作成は 422（`is_success:false` + 前向き案内文 + `requires_subscription:true` 的フラグ）。
- **既存 lat/lng の更新（ラベル変更等）は上限に数えない**（updateOrCreate の更新パスは許可）。
- native: 作成前に件数を見て、上限時は「もっとお気に入りを？釣りログ Plusで無制限」→ /subscription へ**前向きな**導線。サーバ側でも必ずガード（native の抜け道防止）。

**影響範囲**: `RegisterPinController.php`（有効ピン件数チェック）、native `hooks/use-pin.ts` / `api/pin/save.ts` / `components/map/selected-pin-modal.tsx`（上限UI・導線）。`GetPinsController` で現在数を返すと native が上限判定しやすい。

## 3. 自分の現在のプランがわかるようにする

**目的**: 自分が無料か Plus か即分かるように（現状は分かりにくい）。

**現状 [確認済]**
- `subscription.tsx` には現在プラン表示あり（有料は「利用中」バッジ、`currentPlan = myUser.user.plan`）。ただし**課金画面を開かないと分からない**。
- `my-profile.tsx` にプランの常設表示は無い。

**要件 [提案]**
- **プロフィール画面（またはヘッダー）に現在プランを常設表示**（例: 「プラン: 無料」/「プラン: 釣りログ Plus」バッジ）。タップで /subscription。
- `useIsPaid()` / `myUser.user.plan` + `PLAN_LABELS` を利用。有料は期限（`plan_expires_at`）も控えめに出すか検討。

**影響範囲**: native `app/(tabs)/my-profile.tsx`（プラン行追加）、`types/plan.ts` の `PLAN_LABELS` 再利用。

**[決定 2026-07-02]** プロフィール画面に**プラン行**（「プラン: 無料」/「プラン: 釣りログ Plus」）＋タップで /subscription。有料は `plan_expires_at` を控えめに併記。

## 4. インタースティシャル広告のクールダウンを2分にする

**目的**: 表示頻度の調整。

**現状 [確認済]**: `utils/interstitial.ts` `const COOLDOWN_MS = 3 * 60 * 1000;`（3分）。`MIN_DWELL_MS = 15s`（分析滞在の下限）。ポイント非依存・一律・SecureStore で再起動またぎ永続。

**要件 [提案]**: `COOLDOWN_MS = 2 * 60 * 1000;`（2分）に変更。コメントも「2分」に更新。挙動ロジックは不変。

**影響範囲**: native `utils/interstitial.ts` の1定数のみ。

**[要決定]**: なし（値確定済み=2分）。

## 5. 管理画面の修正（MAU / 日別新規獲得 / 課金画面到達・魚種別釣れる度の無料ユーザ日別）

**目的**: 運営KPIの可視化強化。項目6のインプレッションと対。

**現状 [確認済]**
- 管理画面あり（ダッシュボード=リピート率/お気に入り効果、`AdminUserController`、`AdminSubscriptionController`=課金・収益、`funnel` ブランチで課金ファネル table 追加済）。
- `FeatureImpression`（`user_id,type,log_date` 日次ユニーク・staff除外）に `species_rate_gated` を**サーバ側で自動記録済**（funnel ブランチ）。`operation_logs` は `(user_id, action, log_date)` 日次重複排除。

**[決定 2026-07-02]**
- **MAU は「月」単位**（Monthly Active Users）。**横軸=12ヶ月（1〜12月）＋年セレクタ**で各年の月次推移を表示。
  - 定義: ある暦月に `operation_logs` に登場した distinct `user_id`（staff除外・Asia/Tokyo）。月ごと1点。
  - 年ドロップダウンで年を切替（データのある年のみ）。若いアプリのため過去月は空でOK。
- 他の3指標は**日別**（直近30〜90日）で据え置き。

**要件 [提案]**（すべて staff 除外・Asia/Tokyo 日付）
- **MAU（月次・12ヶ月・年セレクタ）**: 上記定義。
- **日別新規獲得数**: `users.created_at` を日別に集計（折れ線/棒）。
- **課金画面到達（無料ユーザ）日別**: `FeatureImpression WHERE type='paywall_view'` を `log_date` 日別ユニークで集計。
- **魚種別釣れる度 表示試行（無料ユーザ）日別**: `type='species_rate_gated'` を `log_date` 日別ユニークで集計。
- 表示は既存の管理画面UI（Chart.js 使用中）に合わせたグラフ＋数値カード。

**影響範囲**: backend `app/Http/Controllers/Admin/*`（ダッシュボード or 専用ページ）、`resources/views/admin/*.blade.php`、`FeatureImpression`（funnel から取り込み）。管理画面のみ=API契約変更なし。

## 6. 課金画面のインプレッションを取得できるようにする

**目的**: 課金画面到達（＝purchase funnel の入口）を計測。項目5の「課金画面到達 日別」の元データ。

**現状 [確認済]**
- funnel ブランチに `POST /api/impressions {type:'paywall_view'}`（`CreateImpressionController` + `CreateImpressionRequest`、`FeatureImpression::record`）が**backend 側は実装済**。
- **native からの送信が未実装**（前回保留にした1行）。

**要件 [提案]**
- native: **無料ユーザーが課金画面（`app/subscription.tsx`）を開いたとき `POST /api/impressions {type:'paywall_view'}` を1回送る**。日次ユニークはサーバ側 `insertOrIgnore` で担保（連打・再訪でも1日1件）。有料/ staff は送らない or サーバで除外（既存は staff 除外済）。
- 送信失敗は握りつぶす（計測が主機能を邪魔しない）。

**影響範囲**: native `app/subscription.tsx`（マウント時に無料なら送信）、`api/impressions/*.ts`（新規クライアント）。契約変更 → `/contract-check`。

**[要決定]**
- 送信トリガーは「画面マウント時」でよいか（推奨）。

## 7〜9. アンケート追加（新規ユーザ / 既存ユーザ / 有料ユーザ）

**目的**
- 新規: どの機能を使ってみたいか（期待・オンボーディング最適化）。
- 既存: どの機能が使いやすいか、欲しい機能・要望。
- 有料: どの有料機能が使いやすいか、欲しい機能・要望。

**現状 [確認済]**
- 既存の**初回アンケート**が1つある: `app/survey.tsx` + `hooks/use-survey.ts` + `POST /api/survey`（`SaveSurveyController`）。
- 保存は **users のカラム**（`survey_source` / `survey_visit_plan` / `survey_frequency` / `survey_answered_at`）。回答するとゲート非表示。
- → **カラム方式は3種の別アンケート（自由記述・要望含む）には拡張性が乏しい**。

**要件 [提案]**
- **回答は専用テーブル `survey_responses`（`id, user_id, survey_type, answers(json), created_at`。`unique(user_id, survey_type)` で1種1回）** に移行/追加。既存の初回アンケート（source/visit_plan/frequency）は現行カラムのまま温存 or 新テーブルへ寄せるかは [要決定]。
- **3種の survey_type**: `new_user_intent` / `existing_user_feedback` / `paid_user_feedback`。
- **表示トリガー・対象セグメント [要決定]**（推奨）:
  - 新規: サインアップ直後のオンボーディング内（初回アンケートと連続 or 統合）。
  - 既存: 既存ユーザー（一定以上の利用 or アプリ起動 N 回）に1回だけモーダル/バナーで依頼。
  - 有料: 購入完了後 or プロフィールから、有料ユーザーに1回。
- **設問形式**: 選択式（機能リスト）＋自由記述（欲しい機能・要望）。機能リストは master でなく設問定義（config か native 定数）で管理。
- **再表示しない**: `survey_responses` に該当 type があればゲート非表示（既存 survey と同じ思想）。

**影響範囲**: backend 新規 `survey_responses` マイグレーション＋モデル、`SaveSurveyController`（type 対応に拡張 or 新コントローラ）、`SaveSurveyRequest`。native `app/survey.tsx`/`hooks/use-survey.ts`（type 別に汎用化）、起動時の出し分け（`components/logged-in-layout.tsx` / `_layout.tsx`）。契約変更 → `/contract-check`。

**[決定 2026-07-02] 目的**: 推測でなく**ユーザーの本音**で「釣りログの本質的な強み/弱み・求められる機能・使いやすさ/使いにくさ」を知る。→ **各アンケートに自由記述を必ず入れる**（定量MC＋定性フリーテキスト）。

**[決定] 保存方式**: 新テーブル **`survey_responses`**（`id, user_id, survey_type, answers(json), created_at`、`unique(user_id, survey_type)`＝type別1回）。既存の初回アンケート（users カラム source/plan/frequency）は**併存**（温存）。

**[決定] 運営が読めること（目的の必須要件）**: 集めるだけでは本音を知れない。**管理画面に `survey_responses` の閲覧を追加**（MCは集計・グラフ／自由記述は一覧表示）。staff除外。

**機能リスト（選択肢の共通ソース・実機能に準拠。休止中のAI戦略は含めない）**
釣行の記録 / 釣果の記録(魚種・サイズ・写真) / フィッシュカード・図鑑 / 釣れる度(時間帯) / 魚種別の釣れる度【Plus】 / 環境データのグラフ(潮汐・天気・水温・風・波) / お気に入りピン / マップで他の人の釣果を見る / 通知

**叩き台 — 設問（各2〜3問・短く）**

`new_user_intent`（新規・オンボーディング内で1回）
- Q1[複数選択] 特に使ってみたい機能は？ → 機能リスト
- Q2[自由記述・任意] 釣りログに期待していること・欲しい機能

`existing_user_feedback`（既存・エンゲージ後に1回）
- Q1[複数選択] よく使う・役に立っている機能は？ → 機能リスト（＝**強み**）
- Q2[複数選択] 使いにくい・分かりにくい機能は？ → 機能リスト＋「特にない」（＝**弱み**）
- Q3[自由記述] 欲しい機能・改善してほしい点（＝**本音**）

`paid_user_feedback`（有料・購入数日後 or プロフィールから1回）
- Q1[複数選択] 契約して良かった・役立っている機能は？ → 魚種別釣れる度 / 広告非表示 / お気に入り無制限
- Q2[自由記述] 有料機能で物足りない・使いにくい点
- Q3[自由記述] 追加してほしい有料機能・要望

**叩き台 — 表示トリガー**
- `new_user_intent`: 既存オンボーディング（`survey.tsx` の source/plan/frequency）の直後に接続、1回のみ。
- `existing_user_feedback`: **エンゲージ済み既存ユーザー**に1回。条件（推奨）＝登録から3日以上 かつ 起動5回以上（既存起動カウンタ流用） かつ 釣行 or 釣果 1件以上。モーダル/バナー、「あとで」可、回答orスキップで再表示なし。
- `paid_user_feedback`: 有料化から数日後（例3日）に1回 or プロフィールから常時アクセス可。有料ユーザーに1回。

**[要確認]** 上記の設問文・選択肢・トリガー条件（起動5回等の数値）でよいか。管理画面の回答閲覧を 1.1.5 スコープに含めるか（含める前提で記載）。

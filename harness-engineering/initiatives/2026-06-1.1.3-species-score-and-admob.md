# 1.1.3 要件定義: 魚種ごと釣れる度（サブスク）＋ AdMob 収益改善

- **Status:** 🟡 要件定義
- **Date:** 2026-06-26
- **対象:** native(tsurilog-native) / backend(tsurilog-backend)。作業は `tsurilog-workspace/` 配下のサブリポのみ。
- **前提/正本:** ADR-0011（ピボット）/ `condition-score-redesign-requirements.md`（釣れる度 再設計＝1.1.2 で実装済み）
- **狙い:** 釣れる度の「深さ（魚種ごと）」に課金を乗せる（ADR-0011 の有料軸）＋ 広告収益を立て直す（250人超・微増中）。

---

## ワークストリーム A：魚種ごと釣れる度（有料サブスク）

### A0. 方針（ADR-0011 と整合）
- **無料＝一般スコア(all)は永久無料**（磁石・1.1.2 実装済み）。**有料＝魚種を選ぶと、その魚に最適化された釣れる度**が見られる。「選んだ瞬間に体感で変わる」こと（relabel にしない）。
- 見せ方は robust（質的・方向性）を維持。精密な符号分解は出さない。

### A1. サブスク（既存 `tsurilog_lite` を流用）
- **製品ID `tsurilog_lite`（プランキー `light`／表示「釣りログ Plus」）をそのまま再利用**（新規 IAP 作成＝再審査を避ける。1.1.1 の審査待ち地獄の回避）。
- **価格 ¥500 → ¥300**（案内表示にも「月額300円」を明記）。
- **付与内容の差し替え**：旧「AIアドバイザー月20回」→「**魚種ごとの釣れる度（見放題）**」。
  - native `types/plan.ts`（priceText/quota/features）、backend `config/ai_strategy.php` の plans ラベル/付与を魚種スコア向けに更新。
- ⚠️ **要検証（外部・記憶で断定しない§8.0.0）**：
  - ASC 上の `tsurilog_lite` の現在ステータス（1.1.1 で審査待ちだった件）。Approved/価格変更可能かを実画面で確認してから進める。
  - **既存¥500課金者の扱い**：旧AI課金者は同じサブで魚種スコアに移行。Apple の自動更新サブ価格変更（値下げ）の既存者適用ルール、サブスク説明文の整合（"AIアドバイザー"記述が残ると審査差し戻し要因）。→ **意思決定が必要（下記 §論点）**。

### A2. backend：魚種ごとスコア
- **基盤は実装済み**：`GetAnalysisRateController` は `fish_id` を受領（現状は一般スコアで無視）。データ較正層 `ConditionDataLayerService` は `fishId` 対応済み（魚種別の釣果で学習可能）。
- **追加するもの**：
  1. **魚種パラメータ table/config**（適水温・適季・潮選好 等の一般生物学ベース）。`config/condition_score.php` に魚種別の general アンカー上書き（例：水温の山の中心を魚種ごとにずらす／月の係数を魚種の旬に合わせる）。`FishMaster` と対応。
  2. `fish_id` 指定時は、その魚種のアンカーで general を計算＋データ層も `fishId` で学習。**選んだ瞬間にスコア分布が変わる**こと。
  3. **プランによるゲート**：`fish_id` 指定の魚種スコアは有料(light/staff)のみ。無料は一般スコア。認可は `auth.apitoken` のユーザのプラン tier で判定（既存の plan 判定を流用）。未課金が魚種スコアを叩いたら `403`＋「サブスク案内」用フラグを返す（native が案内導線を出す）。
- **対象魚種の初期セット（要決定）**：地域主要種（チヌ/クロダイ・メジナ・アジ・シーバス・カサゴ 等）。出典は一般生物学＋釣り情報メディア（要・信頼ソース調査、釣れる度再設計時と同様に WebSearch で裏取り）。

### A3. native：魚種選択 UI ＋ ゲート導線
- **魚種選択 UI**：分析画面（釣れる度）に魚種セレクタを追加（master API から取得・ID ハードコード禁止）。
- **ゲート導線（重要・CV 目的）**：未課金ユーザが魚種を選んで釣れる度を見ようとすると、**スコアの代わりにサブスク案内（釣りログ Plus・月額300円・"魚種ごとの釣れる度が見放題"）を表示**し、プラン画面へ誘導。
- **気づいてもらえる位置に Plus 案内を常設**：分析画面の釣れる度カード付近 or ホームに、控えめだが見える Plus バナー/CTA（¥300 明記）。`SHOW_AI_ADVISOR` のように撤去でなく、魚種スコア用の表示に作り替える。
- **既存課金者**：魚種セレクタがそのまま使える（プラン判定で出し分け）。

## ワークストリーム B：AdMob 収益改善

### B0. 現状診断（コード実測・2026-06-26）
昨日：マッチ率 **4%**、バナー eCPM **¥6**（バナー平均¥30〜80を大幅に下回る）。コードを読んだ結論：
- **`banner-ad.tsx` で `requestNonPersonalizedAdsOnly: true`（常に非パーソナライズ広告を要求）**。→ **fill rate も eCPM も大きく下がる**（パーソナライズ広告は数倍の単価・充足率）。**最有力原因**。
- **同意フローが無い**：UMP（Google のユーザー同意フォーム）未実装、iOS の **ATT（App Tracking Transparency）プロンプト未実装**（app.json に NSUserTrackingUsageDescription / user tracking 設定なし）。→ 同意が取れず、SDK が限定/非パーソナライズに倒れる。
- **`mobileAds().initialize()` / requestConfiguration（maxAdContentRating 等）の明示初期化が見当たらない**。
- 広告ユニット: バナー `7203360950` / ネイティブ `6786093403` / リワード `1588072198`（リワードは旧AI無料回数用＝用途見直し）。本番IDは `EXPO_PUBLIC_APP_VARIANT==="production"` 時のみ（preview/devはテストID＝正常）。

### B1. 施策（ROI 高い順）
1. **同意基盤を入れる（最重要）**：
   - **UMP 同意フォーム**（`AdsConsent` / react-native-google-mobile-ads の consent API）をアプリ起動時に表示・取得。
   - **iOS ATT プロンプト**（`expo-tracking-transparency`、app.json に usage description / SKAdNetwork）。
   - 同意状況に応じて **`requestNonPersonalizedAdsOnly` を撤去**（同意ありはパーソナライズ広告に）。→ fill / eCPM の本命改善。
2. **`mobileAds().initialize()` ＋ requestConfiguration を明示**（起動時、同意後）。
3. **インタースティシャル広告の新設**：自然な遷移（例：分析画面を見終わって戻る／記録保存後 等）に、**頻度上限（例：N分に1回・起動直後は出さない）**を設けて挿入。**磁石(無料の釣れる度)体験を壊さない**範囲で（ADR-0011：無料を劣化させない）。
4. **計測・健全性確認（要・AdMob管理画面＝外部）**：広告ユニットのポリシー状態、地域別 fill、mediation 有無。マッチ率4%は設定要因（NPA/同意）が濃厚だが、ユニットが新しい/制限中の可能性も管理画面で確認。
5. （任意）**mediation 追加**で fill 底上げ（中期）。

### B2. 目標（測定指標）
- マッチ率：4% → **70%+**（同意＋パーソナライズで通常水準へ）。
- バナー eCPM：¥6 → **¥30+**（最低でもバナー平均下限）。
- インタースティシャル導入で 1 セッションあたり収益の底上げ（retention を見ながら頻度調整）。
- 計測：AdMob 管理画面の日次（マッチ率/eCPM/収益）＋リテンション（磁石が壊れていないか）。

### B3. リスク/注意
- **過剰な広告は磁石(無料)を壊す**（ADR-0011）。インタースティシャル頻度は保守的に開始し、リテンション監視で調整。
- ATT/UMP は**ストア審査・各国法令**に関わる（ATT 文言、同意フォームの地域出し分け）。app.json の権限追加はネイティブ指紋変更＝再ビルド必要。
- 広告と有料サブは**トレードオフ**：課金者には広告を減らす/消す設計も検討余地（Plus の価値付け）。

## 確定が必要な論点（実装前に決める）
1. **価格 ¥300**：`tsurilog_lite` を ¥500→¥300 に変更で確定か（ASC 価格変更・既存者への適用方針含む）。
2. **既存¥500課金者の移行**：(a) そのまま¥300扱いに（値下げ）/ (b) 期限まで現状維持 / (c) 個別連絡。サブスク説明文の更新（"AIアドバイザー"→"魚種ごと釣れる度"）。
3. **無料/有料の境界**：無料＝一般スコア(all)、有料＝魚種選択スコア、で確定か（基本の釣れる度は無料据え置き＝ADR-0011）。
4. **対象魚種の初期セット**：どの魚種から出すか（地域主要種）。
5. **AdMob インタースティシャルの挿入箇所と頻度**：どこで・どれくらい許容するか（磁石を壊さない範囲）。
6. **広告 vs 課金**：Plus 課金者は広告非表示にするか。

## スコープ外/先送り
- mediation 本格導入（中期）。リワード広告の用途再設計（AI 無料回数が消えたため）。多季節較正（釣れる度 §8-4）。
